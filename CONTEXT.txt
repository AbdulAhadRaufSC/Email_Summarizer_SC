3. Risks I'd flag (ranked)
R1 — Stale-summary race. [High / correctness] Standard SQS + concurrent workers + single mutable row + auto-publish is last-writer-wins with no ordering guarantee. If email e4's event is processed against a thread of 4 while e5's event (thread of 5) finishes first, the e4 worker overwrites with the older summary and nothing corrects it until e6 arrives — which may be never. The window is timing-dependent and rare at your volume, but it's a real hole and the fix is cheap: a compare-and-set write keyed on a monotonic marker (latest email timestamp or messageId) — UPDATE ... WHERE based_on_email_ts < :new_ts. Bonus: the same guard gives you backfill idempotency and resumability for free. (Alternative: SQS FIFO with MessageGroupId = ticketId — note this still parallelizes across tickets; FIFO enforces per-group order, not global order, so your stated reason for Standard doesn't actually rule it out. I'd take the CAS guard as primary because it protects at the DB boundary regardless of queue and covers the backfill-vs-realtime collision too.)

R2 — Untrusted attachment parsing is an unguarded attack surface. [High / security+stability] Your "self-hosted, so data is safe" framing addresses exfiltration, but the real threat here is malicious files from customer email crashing or exploiting the worker. XLSX/DOCX are zip archives → zip-bomb risk; office XML → billion-laughs entity expansion; PDF/office parsers have a CVE history; a crafted file can OOM or hang the process. Mitigations: run extraction in a resource-limited, sandboxed subprocess/sidecar, enforce hard per-file size + wall-clock timeout, cap decompression ratio, disable external XML entity resolution, cap XLSX rows/cells, and isolate per-file failures to metadata-only fallback so one bad file never fails the ticket. This is the biggest thing missing from your security section.

R3 — "It all fits in one call" is unvalidated. [Medium-High] Email threads re-quote the entire prior chain on every reply, so a 20-email thread is O(n²) text, and an XLSX with thousands of rows can dwarf the conversation. Mitigations: thread normalization / quote-stripping during cleaning (also a quality win — duplicated quoted text degrades LLM output), per-attachment token budgets with truncation, and a graceful-degradation order (drop attachment text first, then oldest quoted history) if the assembled prompt exceeds context. Final sizing depends on the chosen Qwen context length.

R4 — Backfill will flood live traffic, the Email API, and RunPod. [Medium] 40k tickets ≈ ~400k Email API calls plus 40k GPU jobs, bursting into the same queue that serves live tickets. Reuse the worker code, not the live queue: a dedicated backfill queue + rate-limited producer + bounded consumer concurrency, run off-hours, respecting Email API limits. The CAS marker makes it resumable (skip tickets already current). I'll include rough throughput/cost sizing once the model and API limits are known.

R5 — Read-your-writes on the triggering email. [Medium] The event fires because a new email arrived. If there's any propagation delay before that email is in Email_Metadata and retrievable via the API, the worker can summarize a thread missing the very email that triggered it — and persist it as current. Mitigation: verify the triggering messageId is present before summarizing; if absent, let visibility timeout retry rather than proceeding. Needs the API-consistency answer.

R6 — All-or-nothing DLQ can silently bury tickets. [Medium / product] One permanently-unretrievable email sends the whole ticket to the DLQ after max-receives — no summary at all. Given the KB goal, a 19/20 summary with an explicit "1 email unavailable" caveat (status=PARTIAL) is usually more useful than nothing. I'd distinguish transient (retry→DLQ) from permanent per-email failures, and reserve DLQ for total retrieval failure. Either way, alarm on DLQ depth > 0 and write a redrive runbook. This is a decision for you.

Four decisions worth calling out. First, classification is in the schema but not populated by prompt v1 — it reserves the shape so no future redesign is needed, but per the earlier caution that a generative model can't reliably classify into a taxonomy it hasn't been given, population waits for a controlled vocabulary and dedicated logic (a later promptVersion, or a separate classifier). Second, schema completeness is decoupled from model capability: the schema is comprehensive now, but prompt v1 hard-requires only human_summary/customer_issue/current_status and requests the rest best-effort, so guided decoding doesn't over-constrain the 7B model. Third, content.human_summary is denormalized into the summary column so the UI reads it without parsing JSON, while the full document lands in summaryJson. Fourth, this is what makes the schema RAG-ready without a rewrite: executive_summary/customer_issue give clean embedding text, keywords drive hybrid search, current_status and context_completeness are filter facets, and source + schema_version make each JSON self-describing inside the vector index.

On versioning: schema_version pins the structure, promptVersion pins the elicitation (instructions + which fields are actively requested), and modelName/modelVersion pin the generator. Every row carries all four, so any stored summary is fully interpretable and reproducible, and future consumers branch on schema_version to parse summaryJson.

Folder structure
Hexagonal / clean architecture. The dependency rule: domain imports nothing from the outer layers, application depends only on domain, and adapters/entrypoints depend inward — concretes are named in exactly one place, composition.py.

Ports and responsibilities
The ports are Protocols (structural typing → trivial fakes in tests). Data-carrier types live in domain/models.py; shown here by name.

python
# domain/ports.py
from typing import Protocol
from enum import Enum

class EmailMetadataRepository(Protocol):
    def list_email_refs(self, ticket_id: int) -> list[EmailRef]: ...
    # enumerate a ticket's emails from MySQL Email_Metadata, ordered by emailMetaId

class EmailGateway(Protocol):
    def fetch_email(self, message_id: str) -> RawEmail: ...
    # full content + attachments from the internal Email API;
    # raises EmailNotYetAvailable (RYW) or EmailApiTransient

class AttachmentExtractor(Protocol):
    def extract(self, attachment: RawAttachment) -> ExtractedAttachment: ...
    # sandboxed, size/time/ratio-capped; NEVER raises for one bad file —
    # returns EXTRACTED / METADATA_ONLY / FAILED

class ThreadNormalizer(Protocol):
    def normalize(self, emails: list[RawEmail]) -> NormalizedConversation: ...
    # quote-strip, dedup, drop signatures/disclaimers; PII-mask hook lives here

class PromptBuilder(Protocol):
    prompt_version: str
    def build(self, conversation: NormalizedConversation,
              attachments: list[ExtractedAttachment], *, context_budget: int) -> Prompt: ...
    # versioned template + token budgeting (attachment text truncated before history)

class LLMClient(Protocol):
    model_name: str
    model_version: str
    def complete(self, prompt: Prompt, *, json_schema: dict) -> LlmRawResponse: ...
    # one guided-JSON call to RunPod/vLLM; raises LlmTransient on timeout/5xx; returns token usage

class Validator(Protocol):
    def validate(self, raw: LlmRawResponse) -> LlmSummaryOutput: ...
    # parse + schema-validate; raises LlmOutputInvalid -> drives an app-level retry

class WriteMode(str, Enum):
    APPEND_ONLY = "append_only"
    REPROCESS = "reprocess"

class WriteOutcome(str, Enum):
    WRITTEN = "written"
    SKIPPED_SUPERSEDED = "skipped_superseded"

class SummaryRepository(Protocol):
    def get_frontier(self, ticket_id: int) -> int | None: ...
    def upsert(self, ticket_id: int, document: SummaryDocument,
               marker: int, mode: WriteMode) -> WriteOutcome: ...
    # APPEND_ONLY: CAS write only if marker > stored, else SKIPPED_SUPERSEDED
    # REPROCESS:   force-overwrite, frontier := max(stored, marker)
    # the future outbox is a decorator wrapping this port
The WriteMode/WriteOutcome enums put the CAS-and-reprocess decision directly in the type system, so the redrive-vs-administrative distinction can't be violated by accident. Errors are a two-branch hierarchy that the entrypoint maps mechanically to queue behavior: TransientError (EmailApiTransient, EmailNotYetAvailable, LlmTransient) → don't ack, let SQS redeliver; TerminalError (ConversationUnreconstructable, LlmOutputInvalidExhausted) → route to DLQ. Per-attachment failures are not errors — they degrade to METADATA_ONLY/FAILED and flow into context_completeness.

SummarizeTicket.execute(command) orchestrates in order: read the frontier and skip early if command.email_meta_id <= frontier under APPEND_ONLY (saves all downstream work on superseded events); enumerate refs; RYW-verify the triggering message_id is fetchable; fetch all emails with bounded concurrency; extract attachments; normalize; build the budgeted prompt; run the complete → validate loop up to N app-retries (incrementing retry_count); enrich the validated LlmSummaryOutput into a SummaryDocument; and upsert under CAS. WRITTEN → OK/PARTIAL (derived from context_completeness); SKIPPED_SUPERSEDED → ack as a successful no-op; errors propagate to the entrypoint.

The sqs_consumer entrypoint stays thin: parse → build command → execute → map outcome/exception to ack-or-not. One design note for the consumer — in Phase 1, terminal errors ride the natural maxReceiveCount redrive to the DLQ (simple), which costs a bounded number of wasted retries; if that waste matters later, the consumer can short-circuit a known TerminalError straight to the DLQ and ack immediately. backfill.py is the same use-case behind a rate-limited producer on the dedicated queue.
